# Step 06 — Message Persistence (PostgreSQL)

## Overview

Persists every chat message to PostgreSQL so users can retrieve room history after
reconnecting. Follows the repository pattern — the domain layer defines the interface;
the infrastructure layer provides the pgx/v5 implementation. Postgres is **opt-in**:
the server runs fine without it (messages are just not saved).

## Architecture

```
Client ──► ReadPump ──► hub.Broadcast (roomcast + *entity.Message)
                              │
                         Hub.Run()
                         │        │
                      fanout   go persistMessage()
                         │        │
                    WebSocket   Postgres
                              INSERT messages
```

The hub event loop is **never blocked** by the database write — persistence runs in a
dedicated goroutine per message. Errors are logged but do not affect delivery.

## Database Schema

```sql
CREATE TABLE IF NOT EXISTS messages (
    id         TEXT        PRIMARY KEY,
    room_id    TEXT        NOT NULL,
    sender_id  TEXT        NOT NULL,
    type       TEXT        NOT NULL,
    body       TEXT        NOT NULL,
    created_at TIMESTAMPTZ NOT NULL
);

-- Covers all room history queries: filter by room, order by time, paginate.
CREATE INDEX IF NOT EXISTS idx_messages_room_created
    ON messages (room_id, created_at DESC);
```

### Indexing strategy

| Query pattern | Index used |
|---|---|
| `WHERE room_id = $1 AND created_at < $2 ORDER BY created_at DESC LIMIT $3` | `idx_messages_room_created` — single index scan, no sort step |

The composite index `(room_id, created_at DESC)` satisfies the equality filter on
`room_id` and the range + ordering on `created_at` in one pass. No full-table scans.

---

## Message ID

Each message gets a random 16-character hex ID generated by `crypto/rand` at the moment
the client sends it (before broadcast). The same ID is:
- included in the WebSocket frame (`ChatMessage.ID`) for client-side deduplication
- stored as the primary key in Postgres (`ON CONFLICT (id) DO NOTHING` for idempotency)
- returned in history responses

---

## Pagination

`ByRoom` uses **keyset (cursor) pagination** on `created_at`, not offset:

```
GET /rooms/:id/messages?limit=50&before=2025-01-15T10:00:00Z
```

| Parameter | Type | Default | Max |
|---|---|---|---|
| `limit` | integer | 50 | 100 |
| `before` | RFC3339 timestamp | `time.Now()` | — |

Results are returned **newest-first**. To paginate backwards, pass the `created_at` of
the oldest message in the current page as the next `before`.

## API Reference

### `GET /rooms/:id/messages` (JWT-protected)

**Request**
```
GET /rooms/general/messages?limit=20&before=2025-01-15T10:00:00Z
Authorization: Bearer <token>
```

**Response 200**
```json
[
  {
    "id": "a3f9c1e2b8d74f01",
    "room_id": "general",
    "sender_id": "alice",
    "type": "chat",
    "body": "hello",
    "created_at": "2025-01-15T09:59:55Z"
  }
]
```

Returns `[]` (empty array) when no messages match. Newest message first.

**Response 400** — invalid parameters
```json
{ "error": "before must be an RFC3339 timestamp" }
```

**Response 401** — missing or invalid JWT

## Running Locally

```bash
# 1. Start Postgres
docker run -d --name chat-pg \
  -e POSTGRES_DB=chat \
  -e POSTGRES_PASSWORD=pass \
  -p 5432:5432 postgres:16

# 2. Fetch dependencies
cd chat-server
go mod tidy

# 3. Run with Postgres enabled
APP_JWT_SECRET="dev-secret-key-at-least-32-chars!!" \
APP_POSTGRES_ENABLED=true \
APP_POSTGRES_DSN="postgres://postgres:pass@localhost:5432/chat?sslmode=disable" \
go run ./cmd/server

# 4. Get a token
TOKEN=$(curl -s -X POST http://localhost:8080/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"password"}' | jq -r .token)

# 5. Connect and send messages
wscat -c "ws://localhost:8080/ws" -H "Authorization: Bearer $TOKEN"
# {"event":"join","payload":{"room_id":"general"}}
# {"event":"message","room_id":"general","payload":{"message":"hello"}}

# 6. Fetch history
curl -H "Authorization: Bearer $TOKEN" \
  "http://localhost:8080/rooms/general/messages?limit=10"
```

## Performance Notes

| Concern | Approach |
|---|---|
| Write throughput | Async fire-and-forget goroutine — hub never blocks on DB |
| Read latency | Composite index covers the full query (filter + sort + limit) |
| N+1 queries | No ORM — `ByRoom` is a single SQL query |
| DB bottleneck | pgxpool reuses connections; pool size tunable via DSN params |
| Idempotency | `ON CONFLICT (id) DO NOTHING` — safe to replay messages |
